# 多端影音书游记录 App 项目规范（PRD 初版）

## 1. 项目目标

开发一个基于 Flutter 的多端个人记录应用，支持 Android 和 Windows。

应用面向“影视、书籍、游戏”三类内容，核心体验是：

* 用海报墙展示内容
* 用统一的数据模型记录状态、评分、笔记和进度
* 在不同设备之间同步个人数据
* 可接入外部平台同步部分进度与收藏状态
* 在手机和桌面上都保持清晰、顺手、稳定

这个版本先把产品骨架、视觉基线和技术边界定清楚，作为后续开发规范。

---

## 2. 产品定位

### 2.1 产品一句话定义

这是一个面向个人使用的内容记录应用，用来管理“看过什么、看到哪、值不值得记住”，并且在不同设备上无缝接续。

### 2.2 参考方向

参考以下成熟产品或项目的长处，但不直接照搬：

* Trakt：追踪、历史、列表、进度
* Ryot：多媒介统一建模
* MediaTracker：影视/书籍/游戏一体化管理
* Openreads：简洁记录流和跨端体验
* 豆瓣/豆包类内容产品：内容优先、卡片化、信息密度控制

### 2.3 当前版本目标用户

首版默认服务单用户，重点解决个人整理和回看问题，不做社区和社交。

### 2.4 产品体验优先级

面向个人用户时，体验优先级按下面排序：

1. 记录动作足够轻
2. 跨设备续接足够稳
3. 海报墙浏览足够顺
4. 个人数据可控，可备份，可迁移
5. 可接入常用外部平台，减少重复记录

## 4. 核心体验规范

### 4.1 核心体验主线

用户进入应用后，应能顺着下面这条主线完成主要操作：

1. 浏览海报墙
2. 搜索并快捷添加条目
3. 进入详情页
4. 更新状态、评分、笔记、进度
5. 回到自己的收藏、列表和历史中继续管理
6. 换设备后继续使用，不需要重复整理

### 4.1.1 快捷添加主流程

这是首版必须做顺的关键路径。

目标体验：

1. 用户在应用内搜索动漫、书籍、游戏
2. 搜索结果直接带出基础元数据
3. 用户点一次或两次即可完成添加
4. 用户点击“添加”后，先弹出状态选择层
5. 保存后立即出现在本地库中
6. 已绑定 Bangumi 时，把新建记录同步上去

这里的“快捷添加”默认至少带上：

* 标题
* 副标题 / 原名
* 海报或封面
* 年份或发售时间
* 媒介类型
* 基础简介
* 外部来源 ID

首版搜索源采用：

* **Bangumi 单源**

这样做的目的：

* 缩短首版链路
* 降低条目匹配复杂度
* 保证“搜索、添加、同步”使用同一套外部 ID

### 4.2 首页规范

首页必须是内容优先，不是功能菜单优先。

首页至少包含：

* 继续中
* 最近添加
* 最近完成
* 按媒介分类入口
* 自定义列表入口

### 4.3 海报墙规范

海报墙是主视觉，不是附属组件。

必须满足：

* 卡片统一网格
* 图片加载失败有兜底
* 不同屏宽下自动调整列数
* 支持大数据量滚动
* 保持信息层级清楚，避免卡片上堆太多文字

### 4.4 详情页规范

详情页至少包含：

* 封面/海报
* 标题、副标题、年份
* 类型信息
* 简介
* 用户状态
* 用户评分
* 用户笔记
* 当前进度
* 所属列表/标签

### 4.5 跨端连续性规范

这是个人产品的核心能力，不是附加功能。

必须满足：

* Android 和 Windows 使用同一套用户数据
* 用户换设备后，状态、评分、笔记、进度、列表、标签可以继续接上
* 同步状态要可见，但不能打扰主流程
* 同步失败时，用户知道哪里没同步成功
* 本地编辑不能因为远端异常直接丢失
* 首次进入应用时，不强制要求先配置同步

### 4.6 外部平台接入规范

外部平台接入的目标不是替代本应用，而是减少重复录入。

首个优先接入平台：

* Bangumi

Bangumi 官方内容范围不只有动画，还包括：

* 动画
* 书籍
* 音乐
* 游戏
* 三次元

Bangumi 接入长期优先覆盖：

* 条目搜索与匹配
* 收藏状态映射
* 进度同步
* 首次导入已有记录

阶段2 首轮收口：

* 先交付 Bangumi 单源搜索
* 先交付快捷添加后的收藏状态推送
* 先交付状态 / 评分推送
* 剧集 / 页数 / 时长进度推送留到后续 Bangumi follow-up
* Bangumi → 本地的首次导入留到后续任务

首版不做多源聚合搜索。

也就是：

* 应用内搜索先走 Bangumi
* 添加后的外部映射也先绑定 Bangumi subject
* 其他数据源留到后续版本扩展

产品方向按用户确认，长期采用：

* **Bangumi 双向同步**

双向同步首批字段只覆盖：

* 收藏状态
* 观看 / 阅读 / 游玩进度
* 评分

以下字段默认不同步到 Bangumi：

* 私人笔记
* 自定义标签
* 自定义列表

Bangumi 不作为唯一数据源，也不作为本应用的唯一账号体系。

Bangumi 绑定规则：

* Bangumi 不是使用本应用的前置条件
* 用户不绑定 Bangumi，也能完整使用本地记录和跨端同步
* Bangumi 绑定属于增强能力，用于条目搜索、外部收藏同步和进度同步

### 4.7 添加即同步规范

这是当前确定的产品体验。

当用户在应用内搜索并添加一个条目后：

* 本地立即创建记录
* 本地立即保存元数据
* 用户选中的状态立即生效
* 如果用户已绑定 Bangumi，且条目已成功匹配 Bangumi subject
  * 则自动把这条“新建收藏记录”同步到 Bangumi

也就是：

* 在应用里新加的“想看 / 在看 / 已看”
* 对游戏对应的“想玩 / 在玩 / 已玩”
* 应当能作为收藏状态同步上去

如果外部平台同步失败：

* 本地记录仍然保留
* 阶段2 首轮只做轻提示和失败记录
* 显式待同步列表与重试入口留到阶段4 / follow-up

如果本地新条目没有匹配到 Bangumi：

* 允许只保存在本地
* 不阻塞主流程

状态改动的默认规则：

* 用户修改状态后，默认自动同步
* 不要求用户再额外点一次“同步”
* 自动同步失败时，只影响远端，不回滚本地状态

评分与进度的默认规则：

* 用户修改评分后，默认自动同步
* 用户修改进度后，默认自动同步
* 自动同步失败时，只影响远端，不回滚本地评分和进度

## 5. 数据域规范

### 5.1 一等公民媒介类型

首版把以下内容作为一等公民建模：

* 电影 / 剧集
* 书籍
* 游戏

### 5.2 统一实体设计

领域模型先统一，再按媒介补充差异字段。

建议采用以下核心实体：

#### `MediaItem`

表示一个内容对象。

字段方向：

* `id`
* `mediaType`
* `title`
* `subtitle`
* `posterUrl`
* `releaseDate`
* `sourceIds`

#### `MediaDetails`

表示条目详情。

字段方向：

* `overview`
* `genres`
* `creators`
* `runtime / pages / playtime`
* `externalRating`

#### `UserEntry`

表示用户和条目的关系。

字段方向：

* `status`
* `statusLabel`
* `score`
* `review`
* `notes`
* `favorite`
* `reconsumeCount`

#### `ProgressEntry`

表示进度。

字段方向：

* 剧集：季 / 集
* 书籍：页数 / 百分比
* 游戏：时长 / 完成度

#### `ActivityLog`

表示行为时间线。

字段方向：

* `startedAt`
* `finishedAt`
* `pausedAt`
* `abandonedAt`
* `rewatch / reread / replay`

#### `ShelfList`

表示系统列表和自定义列表。

#### `Tag`

表示用户自定义标签。

#### `ExternalAccount`

表示用户绑定的外部平台账号。

字段方向：

* `provider`
* `providerUserId`
* `displayName`
* `connectedAt`
* `lastSyncAt`
* `syncScope`

#### `ExternalMapping`

表示本地条目和外部平台条目的映射关系。

字段方向：

* `provider`
* `localMediaItemId`
* `remoteSubjectId`
* `remoteEpisodeId`
* `lastPulledAt`
* `lastPushedAt`
* `syncDirection`
* `syncStatus`

### 5.3 状态规范

不同媒介共用一套主状态，必要时允许局部扩展。

首版按用户体验拆成面向媒介的展示状态，但内部仍映射到统一主状态。

#### 影视默认显示

* 想看
* 在看
* 已看
* 搁置
* 弃置

#### 书籍默认显示

* 想读
* 在读
* 已读
* 搁置
* 弃置

#### 游戏默认显示

* 想玩
* 在玩
* 已玩
* 搁置
* 弃置

#### 内部统一状态

* `wishlist`
* `inProgress`
* `done`
* `onHold`
* `dropped`

这样可以同时满足：

* 用户看到自然语言状态
* 内部逻辑统一
* 外部平台状态映射更稳定

## 6. MVP 功能范围

### 6.1 本期必须完成

1. 海报墙首页
2. 媒介分类库页
3. 搜索并添加内容
4. 详情页
5. 状态更新
6. 评分和笔记
7. 进度记录
8. 自定义列表
9. 标签
10. 本地持久化
11. 导入导出备份
12. 跨平台数据同步
13. 同步状态提示与失败重试
14. Bangumi 账号绑定
15. Bangumi 双向同步（首批覆盖收藏状态、进度、评分）

### 6.1.1 搜索添加体验要求

搜索添加必须压到低操作成本。

最少满足：

* 搜索结果页直接展示封面、标题、年份、媒介类型
* 用户可在结果卡片上直接点“添加”
* 添加时默认先弹出底部弹层或小弹窗快速选状态
* 保存后不用跳很多页就能继续加下一条

优先交互建议：

* 一次搜索，多次连续添加
* 添加后给出“已加入：想看 / 在看 / 已看”这类即时反馈
* 已在库中的条目要明确展示当前状态，避免重复添加
* 不采用“先默认加入想看 / 想读 / 想玩，再二次修改状态”的默认方案
* 首版搜索结果只来自 Bangumi，先不混入其他来源
* 状态修改默认自动同步，不增加手动同步操作负担
* 评分和进度默认自动同步

### 6.2 第二阶段候选

* Bangumi 账号绑定
* Bangumi 首次导入
* Bangumi 双向同步扩展字段
* 第三方数据源导入
* 推荐内容
* 日历和上线提醒
* 桌面组件 / 通知

### 6.3 当前明确不做

* 社交关系
* 评论社区
* 公共主页
* 实时协作
* 插件系统

## 7. 视觉与设计规范

### 7.1 设计总原则

这不是纯内容管理工具，也不是纯展示型应用。

设计上要同时满足：

* 内容优先
* 信息清楚
* 海报有氛围
* 长时间使用不累
* Android 和 Windows 都不违和

### 7.2 视觉基线

首版采用：

**Material 3 作为结构基线，局部加入玻璃质感，不走全局 iOS 玻璃风。**

### 7.3 为什么不直接走纯 iOS 玻璃规范

原因如下：

* 应用首发平台不含 iOS
* 海报墙本身已经是高视觉密度界面
* 大面积模糊会拉高渲染开销
* Android 上全局 iOS 化容易显得不协调
* Windows 更适合把材质效果放在窗口层或少量辅助层

### 7.4 玻璃效果使用边界

允许使用玻璃/半透明材质的区域：

* 顶栏
* 浮层
* 弹窗
* 详情页头图叠层
* 小面积工具条

禁止直接使用玻璃/重模糊的区域：

* 整页背景大面积持续模糊
* 整个海报墙底板
* 长列表滚动区域
* 主要内容区的全屏毛玻璃

### 7.5 Windows 端设计原则

Windows 端优先做：

* 更宽的栅格
* 左侧导航或双栏布局
* 窗口级材质适配
* 鼠标悬停反馈
* 快捷键和右键菜单预留

不要把移动端 UI 直接拉宽后硬搬到桌面。

### 7.6 Android 端设计原则

Android 端优先做：

* Material 3 导航模式
* 单手操作优先
* 滚动与返回手势自然
* 浮层和底部操作区清楚

## 8. 技术规范

### 8.1 技术栈

首版技术选型：

* Flutter
* `flutter_riverpod`
* `go_router`
* `drift`

### 8.2 选型原则

#### Flutter

原因：

* Android 和 Windows 可共用一套 UI 与业务逻辑
* 适合做高质量卡片、动效和自适应布局

#### Riverpod

原因：

* 适合异步状态
* 适合搜索、分页、缓存、筛选
* 比较适合中大型 Flutter 项目后续扩展

#### GoRouter

原因：

* 路由表达清楚
* 适合多端导航和未来深链接扩展

#### Drift

原因：

* 适合结构化本地数据
* 适合列表、标签、历史、进度这类关系数据
* 比轻量 KV 更适合这个项目

### 8.3 数据策略

首版建议采用：

**本地优先 + 正式云同步能力。**

也就是：

* MVP 默认本地可完整使用
* 首次进入默认本地模式
* 用户后续可自行配置同步目标后启用跨设备同步
* 本地数据始终可回退、可备份
* 应用负责多设备合并和同步分发逻辑
* 首版不强依赖应用自建账号系统

### 8.4 外部同步策略

外部平台同步和应用内云同步必须分开设计。

规则如下：

* 应用内跨端同步负责 Android 和 Windows 之间的数据一致性
* Bangumi 负责外部导入与部分字段同步
* 本地库始终是应用运行时主数据源
* 外部平台字段需要单独做映射，不直接污染内部模型
* 笔记、标签、自定义列表默认以本应用为准
* 本地新建的收藏记录，在满足匹配条件时可自动推送到 Bangumi
* 首版搜索数据源与首版同步数据源统一使用 Bangumi
* 应用内跨端同步不依赖 Bangumi 账号体系

### 8.5 跨平台同步协议选择

首版默认支持以下两类同步目标：

* `WebDAV`
* `S3-compatible`

默认展示顺序：

1. `WebDAV`
2. `S3-compatible`

默认启用方式：

* 首次进入应用，不强制要求立刻配置同步
* 设置页提供同步配置入口
* 用户完成配置后，再启用跨设备同步

这样安排的原因：

* 对国内个人用户，`WebDAV` 更容易找到现成服务
* `S3-compatible` 更适合承载稳定的数据对象同步
* 两者都比“必须先注册应用自家云账号”更符合当前项目方向

### 8.5.1 WebDAV 作为默认入口的判断

`WebDAV` 在国内环境下，算是“可用且相对友好”，但不是“零门槛”。

友好的地方：

* 国内已有现成服务支持 `WebDAV`
* 用户不一定要自建服务器
* 对熟悉同步工具的用户，上手成本可接受

不够友好的地方：

* 仍然需要用户自己准备 `WebDAV` 账号或服务
* 不同服务商限制差异较大
* 文件数、请求频率、上传大小等限制会直接影响同步体验

因此产品上应这样处理：

* `WebDAV` 放在首位，作为国内推荐选项
* 首次配置时提供常见服务商预设
* 不把 `WebDAV` 描述成“免配置”

### 8.5.2 S3-compatible 作为默认选项的判断

这里指的是 `S3-compatible`，不限定为 AWS S3 本体。

原因：

* 国内云厂商已有兼容 S3 API 的对象存储
* 协议层更适合做对象化同步
* 后续扩展到自托管 MinIO、腾讯 COS、阿里 OSS 更自然

产品上应这样处理：

* 文案写 `S3-compatible`
* 预设常见目标类型，例如 MinIO、腾讯 COS、阿里 OSS
* 不把 AWS S3 中国区当成唯一默认实现

### 8.5.3 实现边界

同步目标可以是 `WebDAV` 或 `S3-compatible`，但同步逻辑必须由应用自己控制。

也就是：

* 不直接双向同步整个数据库文件
* 不把远端存储当作“自动帮你解决冲突”的黑盒
* 应用内部要有自己的同步队列、版本标记、冲突处理和重试机制

### 8.5.4 主同步模型

首版同步模型采用：

* **对象 / 日志同步作为主同步**
* **快照包同步只用于备份恢复**

区别如下：

* 对象 / 日志同步：按单条对象或单次变更增量同步
* 快照包同步：按整包数据导出、上传、恢复

首版落地要求：

* 日常跨设备同步走对象 / 日志同步
* 导入导出、手动备份、灾难恢复走快照包
* 不允许用快照包覆盖日常自动同步链路

### 8.5.5 对象同步最小要求

首版不要求做完整事件溯源，但至少要有：

* `updatedAt`
* `deletedAt`
* `syncVersion`
* `deviceId`
* `lastSyncedAt`

用于支持：

* 增量上传
* 增量拉取
* 冲突判断
* 重试恢复

### 8.5.6 冲突处理策略

首版冲突处理采用：

* **最后修改优先**
* **保留冲突副本**

规则如下：

* 同一字段发生冲突时，默认以最后修改时间较新的值生效
* 重要文本字段发生冲突时，除了采用较新的值，还要保留一份冲突副本
* 冲突副本不能静默丢失
* 用户后续可以手动合并或清理冲突内容

首版优先保留冲突副本的字段：

* 短评
* 私人笔记

首版可直接按最后修改覆盖的字段：

* 状态
* 评分
* 进度
* 收藏时间类元信息

## 9. 架构规范

### 9.1 分层原则

* 展示层

* 领域层
* 数据层

* `features/library`

* `features/settings`
* `shared/ui`
* `shared/theme`
* `shared/domain`
* `shared/data`

### 9.3 平台差异处理原则

平台差异只放在必要位置：

* 布局层
* 窗口层
* 输入交互层

业务规则、数据结构、状态流不要按平台复制。

### 9.4 长期架构契约归档规则

长期生效、影响多个阶段或多个子任务的规则，不只写在某个子任务 PRD 中。

归档顺序如下：

1. 根父任务 PRD：产品方向、阶段拆分、跨阶段边界
2. 阶段父任务 PRD：本阶段所有子任务共享的接口和范围边界
3. `.trellis/spec/architecture/`：跨层架构契约、local-first 同步契约、任务树治理规则

* `.trellis/spec/architecture/system-boundaries.md`
* `.trellis/spec/architecture/local-first-sync-contract.md`

子任务可以复述父任务规则用于执行，但不能成为唯一事实来源。
如果子任务修改共享签名、同步边界、provider 归属或错误策略，必须同步更新阶段父任务 PRD 和对应 spec。

## 10. 交互规范

### 10.1 记录动作必须轻

用户最常做的动作必须少步骤：

* 改状态
* 打分
* 写短笔记
* 改进度
* 加入列表

### 10.2 搜索与添加规范

搜索添加必须做到：

* 快速返回结果
* 支持按媒介筛选
* 条目卡片足够简洁
* 点“添加”后先选状态，再保存
* 添加后立即可进入详情或更新状态
* 状态变更后默认自动同步

### 10.2.1 自动同步交互规范

自动同步是默认行为，但不能打扰记录动作本身。

必须满足：

* 状态、评分、进度修改后立即进入同步队列
* UI 优先反馈“已保存到本地”
* 远端同步单独展示为轻提示，不阻塞用户继续操作
* 阶段2 首轮失败只保留轻提示和失败记录，不回滚本地
* 显式“待同步”状态和重试入口留到阶段4 / follow-up

### 10.3 空状态规范

首版必须认真设计空状态。

至少覆盖：

* 没有任何记录
* 某个分类为空
* 搜索无结果
* 列表为空

## 11. 性能规范

### 11.1 首版性能底线

必须优先保证：

* 海报墙滚动流畅
* 切页稳定
* 图片缓存可控
* 大量条目时不卡顿

### 11.2 性能风险点

重点避免：

* 大面积 `BackdropFilter`
* 卡片上叠太多实时效果
* 无限制图片解码
* 长列表不做分页或虚拟化思路

## 14. 当前结论

当前项目规范结论如下：

1. 用 Flutter 做 Android + Windows 双端
2. 产品核心是“海报墙 + 记录流 + 跨设备续接”
3. 用统一模型覆盖影视、书籍、游戏
4. 视觉上用 MD3 打底
5. 玻璃质感只做局部强化，不做全局风格
6. 数据层走本地优先，同时正式支持跨平台同步
7. Bangumi 作为首个外部接入平台，优先同步收藏状态和进度
8. 先做个人用户 MVP，再逐步扩展
9. 首版搜索源采用 Bangumi 单源，不做多源聚合
10. 首版默认支持 WebDAV 和 S3-compatible 作为同步目标
11. 国内默认推荐 WebDAV，进阶用户可选 S3-compatible
12. 主同步走对象 / 日志同步，快照包只做备份恢复
13. 冲突处理采用最后修改优先，并为文本字段保留冲突副本
14. Bangumi 绑定是可选增强，不是应用前置登录条件
15. 首次进入默认本地模式，同步在设置页按需开启

## 15. 分阶段实施与子母任务拆分

### 15.1 拆分原则

当前 PRD 可以落地，但不能把所有能力一次并行开做。

拆分原则如下：

* 先做本地可用，再做外部接入
* 先打通主链路，再补同步运维
* 先做用户直接感知的能力，再做扩展项
* 每个阶段都要能形成一个可验证的中间结果

### 15.2 推荐执行顺序

推荐按下面顺序推进：

1. 阶段1：本地可用的核心骨架
2. 阶段2：Bangumi 搜索、快捷添加与同步
3. 阶段3：跨端同步内核（WebDAV / S3-compatible）
4. 阶段4：设置、备份恢复与同步运维
5. 阶段5：桌面适配、性能与验收

### 15.3 子母任务结构

父任务：

* `04-18-flutter-media-tracker`

子任务：

* `04-19-phase1-local-core`
* `04-19-phase2-bangumi-flow`
* `04-19-phase3-sync-engine`
* `04-19-phase4-settings-backup`
* `04-19-phase5-polish-qa`

### 15.4 阶段目标

#### 阶段1：本地可用的核心骨架

目标：

* 把应用做成“不开任何外部能力也能用”的本地版本

覆盖：

* Flutter 工程骨架
* 主题、路由、自适应导航壳
* 本地数据表与仓储层
* 首页、库页、详情页
* 本地状态、评分、笔记、进度、列表、标签

交付结果：

* 用户首次进入后，不配置同步、不绑定 Bangumi，也能完成完整记录流程

#### 阶段2：Bangumi 搜索、快捷添加与同步

目标：

* 打通“Bangumi 搜索 → 快捷添加 → 自动同步”的主体验

覆盖：

* Bangumi 搜索接入
* 条目映射与元数据入库
* 快捷添加交互
* Bangumi 可选绑定
* 状态、评分同步
* 远端进度推送留到后续 Bangumi follow-up

交付结果：

* 主链路可用，用户可以从 Bangumi 搜索结果一键加入并同步收藏状态与评分

#### 阶段3：跨端同步内核（WebDAV / S3-compatible）

目标：

* 建立 Android 与 Windows 之间的数据一致性能力

覆盖：

* 对象 / 日志同步模型
* 同步队列
* 冲突判断
* WebDAV 适配器
* S3-compatible 适配器

交付结果：

* 两台设备之间可以稳定做增量同步

#### 阶段4：设置、备份恢复与同步运维

目标：

* 把同步能力从“能跑”收成“能配、能看、能修”

覆盖：

* 设置页同步入口
* 连接测试
* 备份导入导出
* 待同步列表
* 冲突副本查看与重试

交付结果：

* 用户可以独立完成同步配置、备份恢复和失败处理

#### 阶段5：桌面适配、性能与验收

目标：

* 把产品从“功能可用”收成“日常可用”

覆盖：

* Windows 布局与交互优化
* 海报墙性能
* 图片缓存与滚动性能
* 空状态、错误状态、回归测试

交付结果：

* Android 与 Windows 都达到可长期使用的质量线

### 15.5 依赖关系

依赖关系如下：

* 阶段1 是后续所有阶段的基础
* 阶段2 依赖阶段1
* 阶段3 依赖阶段1，和阶段2 可部分并行，但联调要在阶段2后进行
* 阶段4 依赖阶段2与阶段3
* 阶段5 依赖前面全部阶段

### 15.6 里程碑建议

建议按下面里程碑判断是否继续扩展：

* M1：本地版可单独成立
* M2：Bangumi 主链路成立
* M3：跨端同步成立
* M4：设置、备份、冲突运维成立
* M5：桌面适配与性能达标

### 15.7 范围控制建议

如果开发资源有限，建议这样收口：

* 首个可发布版本：阶段1 + 阶段2
* 次版本：阶段3 + 阶段4
* 稳定版：阶段5

### 15.8 任务树治理规则

任务树保持产品和技术交付导向。

规则如下：

* 规范补强、父任务吸收、spec 落盘，不单开新的兄弟任务
* 只有出现新的用户可感知能力，或可独立执行的技术交付片，才新增 child task
* 阶段父任务承接共享契约，child task 只承接实现范围、验收标准和局部研究记录
* 同一个规则同时影响 2 个以上 child task 时，先更新阶段父任务，再更新 spec
* 只影响当前 child task 的细节，留在该 child PRD，不上提到根父任务
